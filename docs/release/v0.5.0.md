# iqdb-types v0.5.0 — API Freeze

**Locked for 1.x.** v0.5.0 makes the deliberate, conservative commitment the whole crate has been building toward: the public API is **frozen**. The complete frozen surface is recorded in `dev/ROADMAP.md`; from here only additive, non-breaking changes are allowed before 2.0. Every other iQDB crate will depend on this vocabulary, so the freeze is the careful step — taken only after the surface was property-tested, consumer-validated, and verified across platforms and the MSRV.

## What is iqdb-types?

The root of the iQDB dependency graph — the shared vocabulary every other `iqdb-*` crate speaks: `Vector`, `VectorId`, `DistanceMetric`, `Metadata`/`Value`, `Filter`, `SearchParams`, `Hit`, and the `IqdbError` domain error. Pure data: no engine, no storage, no I/O, one runtime dependency (`error-forge`).

## What's new in 0.5.0

### The API is frozen for 1.x

The full public surface is recorded in `dev/ROADMAP.md` under *"Frozen public API (1.x)."* The commitment: additive, non-breaking changes (new methods, new variants on the `#[non_exhaustive]` enums) remain allowed across 1.x; anything else waits for 2.0.

The deliberate freeze decisions:

- **`VectorRef` retained.** The real consumers pass `&[f32]` directly, but a clean zero-copy borrowed view is worth keeping for the Database/RAG/query-builder layers — and re-adding it later would be fine, while removing it later would be a 2.0 break. Conservative call.
- **`DistanceMetric` and `IqdbError` are `#[non_exhaustive]`.** The metric set and the error set can grow within 1.x without breaking consumers (who already write a wildcard arm).
- **`Value` and `Filter` stay exhaustive.** Consumers construct *and* exhaustively match these, so full matchability is worth more than additive growth.
- **`SearchParams` and `Hit` are plain public-field structs.** This keeps the Tier-1/Tier-2 struct-update ergonomics (`SearchParams { ef: Some(..), ..SearchParams::new(k, metric) }`). Per-index tuning knobs (IVF `nprobe`, PQ rerank, …) belong in each index crate's own config, **not** in the shared `SearchParams`, so this struct is expected to stay stable.

### Cross-platform verification

The surface was verified on **Windows and Linux (WSL2 Ubuntu)**, each on **both** the stable toolchain and the **1.87 MSRV** — clippy, the full test suite, and docs all clean on every combination. macOS is covered by the CI matrix.

## Breaking changes

**None.** This release changes no public API; it records the freeze and adds cross-platform verification.

## Verification

Run on Windows x86_64 **and** Linux x86_64 (WSL2 Ubuntu), each on stable (1.95.0) and the 1.87 MSRV (`cargo +1.87`). The same commands run in the CI matrix across Linux, macOS, and Windows at stable and 1.87:

```bash
cargo fmt --all -- --check
cargo clippy --all-targets -- -D warnings
cargo clippy --all-targets --all-features -- -D warnings
cargo test
cargo test --all-features
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --all-features
cargo deny check
cargo audit
```

All green on every platform/toolchain combination. Test counts: **77** default, **88** `--all-features`. `cargo deny check` reports `advisories ok, bans ok, licenses ok, sources ok`; `cargo audit` finds zero advisories.

## What's next

- **v0.6.0 → v0.9.x — Alpha / Beta / RC.** With the API frozen, the remaining work is hardening and polish: bug-fixes, doc refinement, and any final issues real integration surfaces — no API changes. Then **v1.0.0**.
